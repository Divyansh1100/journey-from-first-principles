# Session 06 — From One Computer to a Scalable System

## Part 2A — The First Bottleneck

---

# Reasoning Journey

At the end of the previous stage, one important conclusion had already been reached.

A personal laptop was perfectly capable of serving a very small number of users.

However, the same architecture immediately began showing signs of weakness as the platform started growing.

Instead of accepting this as a limitation, the discussion shifted towards understanding **why** the existing system could no longer satisfy the increasing demand.

Rather than asking,

> **"What technology should we use?"**

the discussion began with a much simpler question.

> **"What exactly is my laptop doing?"**

This question appears simple, yet it gradually revealed several hidden responsibilities that a single machine performs while operating a website.

---

# Understanding the Responsibilities of a Single Computer

Initially, the following workflow was proposed.

A user uploads a video through the website.

The platform receives the uploaded file.

The system processes it by assigning an appropriate location, storing its metadata and preparing it for future retrieval.

Whenever another user searches for that video, the platform locates the stored information and delivers a copy of the original content to the requesting user.

The simplified workflow appeared as follows.

```text
User Uploads Video
        │
        ▼
Platform Receives Video
        │
        ▼
Store Video & Metadata
        │
        ▼
User Requests Video
        │
        ▼
Retrieve Stored Information
        │
        ▼
Deliver Video
```

For a small platform, this process appeared perfectly reasonable.

The same computer performed every task.

No additional infrastructure was necessary.

---

# A Small Refinement — Receiving Is Not Downloading

During the discussion, an important distinction emerged.

Initially, the workflow was described as:

> "The laptop downloads the uploaded video."

Although understandable, this wording was carefully examined.

A question was raised.

> **Is the laptop really downloading the video?**

After reflecting upon the direction of information flow, the understanding became clearer.

When a user uploads a file, the platform is **receiving** information from the user.

The platform has not requested an already existing file from another computer.

Instead, another device is sending information directly to it.

This process is therefore more accurately described as **receiving an upload**.

Downloading represents a different situation.

Suppose a file already exists somewhere else.

The platform now requests that existing file and copies it to its own system.

Only in this case is the system downloading information.

The distinction can be illustrated as follows.

### Receiving

```text
User

│
│ Upload
▼

Platform
```

The platform receives information that is being sent for the first time.

---

### Downloading

```text
Existing Server

│
│ Download Request
▼

Platform
```

The platform requests information that already exists elsewhere.

Although both involve transferring data, the direction and purpose of communication are different.

This refinement may appear small, yet it illustrates an important engineering habit.

Engineers pay careful attention not only to **what** information moves, but also **how** and **why** it moves.

Precise terminology often reflects precise thinking.

---

# Growth Changes Everything

The original architecture functioned effectively because only a handful of people interacted with the platform.

Then a new situation was imagined.

Instead of ten users...

Imagine ten thousand.

Soon after...

Imagine ten million.

At first glance, it might appear that the platform is simply repeating the same tasks more frequently.

However, this assumption quickly proves inaccurate.

The nature of the workload itself changes.

The system is no longer performing one activity at a time.

It is performing many independent activities simultaneously.

For example, at any given moment:

* One user may be uploading a new video.
* Another may be watching an existing video.
* Someone else may be searching for content.
* Another person may be commenting.
* Thousands of users may be pressing the "Like" button.
* New accounts may be registering.
* Existing users may be logging in.

The platform is therefore no longer executing a single sequence of operations.

Instead, countless independent events occur continuously.

This realization significantly changed the perspective of the discussion.

The problem was not simply that the laptop had become slower.

The real problem was that the responsibilities assigned to a single machine had increased dramatically.

---

# The First Bottleneck

At this stage, two major limitations naturally emerged.

## Storage

Every uploaded video occupies physical storage.

As the number of users increases, the collection of videos grows continuously.

Eventually, the storage capacity of a personal laptop becomes insufficient.

---

## Processing

Storage alone does not solve the problem.

Every user interaction requires computation.

Searching.

Uploading.

Processing.

Retrieving.

Delivering.

Managing user accounts.

Updating metadata.

Responding to millions of requests.

All of these activities compete for the same processor.

Even an extremely fast processor has practical limits.

Eventually, requests begin waiting.

Response times increase.

The platform becomes unreliable.

---

# First Refined Understanding

An important realization emerged from this discussion.

Initially, the laptop appeared to perform one simple responsibility.

Upon closer examination, it was discovered that the same machine was simultaneously acting as:

* A storage device.
* A processing unit.
* A request manager.
* A communication endpoint.
* A content delivery system.

The issue therefore was not merely insufficient hardware.

The issue was that **one machine had been assigned too many responsibilities.**

This observation naturally led to the next engineering question.

> **Should we continue making one computer stronger...**

> **...or should we begin sharing these responsibilities among many computers?**

The answer to that question becomes the focus of the next stage of the session.
